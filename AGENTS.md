# Repository Guidelines

## プロジェクト構成

このリポジトリは OpenTofu ベースのインフラ管理用です。サービスごとのルートモジュールは基本的に `<インフラ名>/<サービス名>` というディレクトリ構成で配置してください。例: `proxmox/alloy`。各サービスディレクトリには `provider.tf`、`main.tf`、必要に応じて責務別の `*.tf` を置きます。複数サービスで再利用する共通定義は `modules/` 配下に切り出し、ルートモジュール側から参照してください。`.terraform.lock.hcl` は各ルートモジュールのロックファイルとしてコミット対象です。

## 開発・検証コマンド

このプロジェクトでは `terraform` ではなく `tofu` を使います。ツールのバージョンはリポジトリルートの `mise.toml` で `opentofu = "1.11.6"` に固定されています。

- `tofu init`: provider 取得と backend 初期化を行います。
- `tofu fmt -recursive`: HCL の整形をそろえます。
- `tofu validate`: 構成の静的検証を行います。
- `tofu plan`: 変更差分を確認します。
- `mise install`: `tofu` が未導入の環境でツールをそろえます。

機密情報が必要な操作では、必要に応じて `fnox.toml` 経由で 1Password の secrets を利用します。

## コーディング規約

HCL の整形は `tofu fmt` の結果を正とします。インデントは 2 スペース、1 行 1 引数を基本にし、長い定義は適切に改行してください。リソース名、`locals`、`output` は小文字の snake_case を使います。例: `proxmox_vm_qemu.alloy_node`。ファイル名は環境名ではなく役割ベースで付けます。

## テスト方針

専用の自動テストはまだありません。変更ごとに最低限 `tofu fmt -recursive`、`tofu validate`、`tofu plan` を実行してください。リソース変更を含む PR では `tofu plan` の要点を記載し、認証や backend 制約で実行できない場合は理由を明記してください。

## コミット

履歴は Conventional Commits 形式に従うこと。

- 形式: `<type>: <summary>`
- コミットメッセージは件名・本文ともに日本語で記載すること（Conventional Commits の `type` は英語のままで可）。
- 3行目以降には具体的な変更内容を記載すること。
- 複数行のコミットメッセージを作成する場合は、コミットのコマンドの実行方法によってはうまく複数行になっていない場合がある。コミット後に必ず確認し、複数行になっていない場合は修正すること。
- コミットメッセージ本文に改行を入れるときは、`-m` 引数内に `\n` を書かないこと。`git commit -m "<件名>" -m "<本文>"` のように `-m` を分けるか、メッセージファイルを使うこと。
- 1コミット1目的を徹底すること。
- 適切な粒度でコミットを行うこと。
- GPG署名を無効化しないこと。署名関連でコミットエラーになった場合はユーザーに報告すること。

## セキュリティと設定

認証情報、ローカル state、秘密情報入りの `*.tfvars` はコミットしないでください。`fnox.toml` には 1Password 参照が定義されているため、秘密値は環境変数や安全な secrets 管理経由で扱います。state のキーや backend 設定を変更する場合は影響範囲を確認してからマージしてください。
